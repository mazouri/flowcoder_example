## 1. 产品概述

### 1.1 一句话描述

灵薯智作是一款面向小红书内容创作者和运营团队的 **桌面端专业工具**，提供"采集→拆解→创作→发布→数据分析"的全链路自动化运营能力。

### 1.2 目标用户

| 用户角色 | 典型需求 |
|---------|---------|
| 个人创作者 | 快速找到爆款灵感、AI 辅助写文案、一键发布 |
| MCN 机构 / 运营团队 | 多账号矩阵管理、批量采集素材、数据看板 |
| 电商带货博主 | 爆款拆解、仿写高转化文案、定时发布 |
| 品牌方市场部 | 竞品监控、博主分析、内容批量生产 |

### 1.3 平台支持

- Windows 10 及以上（64 位）
- macOS 10.15 及以上（Intel / Apple M1/M2/M3/M4 芯片）

### 1.4 核心价值主张

- 覆盖小红书运营全流程，从选题到发布一站式完成
- 本地优先 (Local-First) 数据架构，用户数据不上传服务器
- 非侵入式 RPA 仿真技术，模拟真人操作，降低封号风险
- 集成主流大模型 API（火山引擎豆包、阿里百炼通义千问）

## 2. 核心功能模块拆解

### 2.1 一键采集素材

**功能描述：** 自动从小红书网页端抓取笔记内容，包括图片、视频、文案、互动数据等，自动去水印，毫秒级抓取。

| 子功能 | 说明 |
|--------|------|
| 一键采集笔记 | 输入笔记链接或关键词，批量抓取笔记内容 |
| 一键采集博主信息 | 抓取博主主页资料、粉丝数、互动数据 |
| 一键采集博主笔记 | 批量抓取指定博主的所有/部分笔记 |
| 自动去水印 | 抓取时自动去除图片/视频水印 |
| 素材库管理 | 采集的素材统一存入本地素材库，支持分类、搜索 |

**交互流程：**
```
用户输入关键词/链接 → 系统自动在内置浏览器中加载页面
→ Extension 插件解析页面 DOM → 提取图片/文案/数据
→ 自动去水印处理 → 存入本地数据库 → 表格化展示
```

### 2.2 AI 深度拆解

**功能描述：** 输入爆款笔记链接，AI 透视笔记的内容结构、情绪钩子与底层逻辑，还原爆款公式。

| 分析维度 | 说明 |
|----------|------|
| 标题拆解 | 分析标题的结构模式（数字法、疑问法、痛点法等） |
| 内容结构 | 拆解正文段落的逻辑框架（开头钩子、中间价值、结尾 CTA） |
| 情绪分析 | 识别文案中的情绪触发点和共鸣元素 |
| 选题分析 | 归纳爆款选题方向和切入角度 |
| 爆款因子 | 综合分析为什么这篇笔记能成为爆款 |

**交互流程：**
```
用户粘贴爆款笔记链接 → 系统采集笔记全部内容
→ 调用大模型 API 进行结构化分析
→ 以可视化卡片形式展示分析结果
→ 可一键生成仿写文案
```

### 2.3 AI 智能仿写

**功能描述：** 基于拆解结果，AI 自动生成高质量仿写文案，保留爆款基因但内容完全原创。

- 保留原文的结构模式和情绪节奏
- 替换具体内容为用户指定的主题/产品
- 支持多次迭代和风格调整

### 2.4 AI 辅助原创

**功能描述：** 从零开始创作原创内容，AI 提供选题建议、大纲生成、正文撰写等辅助。

### 2.5 AI 灵感裂变

**功能描述：** 基于一个核心主题或爆款笔记，AI 裂变出多个不同角度的内容方向，实现"一变多"的矩阵内容生产。

### 2.6 博主分析

**功能描述：** 深度拆解头部博主，分析其内容策略、发布规律、爆款比例等，复制爆款基因。

| 分析维度 | 说明 |
|----------|------|
| 账号概览 | 粉丝数、获赞数、笔记数等基础数据 |
| 内容分析 | 笔记类型分布、爆款率、更新频率 |
| 发布规律 | 发布时间段、频率分析 |
| 互动分析 | 点赞/收藏/评论比例和趋势 |

### 2.7 矩阵发布

**功能描述：** 多账号一键分发，支持扫码登录和定时自动发布。

| 子功能 | 说明 |
|--------|------|
| 多账号管理 | 管理多个小红书账号，支持扫码登录 |
| 一键发布 | 编辑好的笔记一键发布到指定账号 |
| 定时发布 | 设置发布时间，系统自动在指定时间发布 |
| 批量分发 | 同一内容或不同内容分发到多个账号 |
| 电脑端自动发布 | 无需手机操作，电脑端直接完成发布 |

**交互流程：**
```
创作/编辑笔记 → 选择发布账号（支持多选）
→ 设置发布时间（立即/定时） → 确认发布
→ 系统通过 RPA 在内置浏览器中模拟操作完成发布
→ 发布状态实时反馈
```

### 2.8 全域矩阵智能看板

**功能描述：** 整合采集、创作和发布全流程数据，AI 智能分析，全局掌控流量趋势。

| 看板维度 | 说明 |
|----------|------|
| 总览看板 | 全账号汇总数据，核心指标一目了然 |
| 数据看板 | 各账号的详细运营数据、趋势图 |
| 内容效果分析 | 各笔记的互动数据、爆款概率分析 |
| AI 智能分析 | AI 自动生成数据洞察和优化建议 |

## 6. 关键技术实现详解

### 6.1 一键采集素材 — 详细技术实现方案

#### 6.1.1 整体架构

一键采集的核心是 **Electron 内嵌 Chromium + Chrome Extension Content Script + 网络请求拦截** 三层协同工作：

```
┌────────────────────────────────────────────────────────────────────┐
│                        Electron 主进程                              │
│                                                                    │
│  ┌──────────────┐   ┌──────────────┐   ┌────────────────────────┐ │
│  │  采集任务      │   │  数据处理      │   │  持久化存储              │ │
│  │  调度器        │   │  管线          │   │                        │ │
│  │              │   │              │   │  SQLite (better-sqlite3) │ │
│  │ - 任务队列    │   │ - 去水印      │   │  + 本地文件系统           │ │
│  │ - 并发控制    │   │ - 数据清洗    │   │                        │ │
│  │ - 错误重试    │   │ - 格式归一    │   │                        │ │
│  └──────┬───────┘   └──────┬───────┘   └────────────┬───────────┘ │
│         │                  │                        │             │
│         │     IPC (ipcMain / ipcRenderer)            │             │
│─────────┼──────────────────┼────────────────────────┼─────────────│
│         │                  │                        │             │
│  ┌──────▼──────────────────▼────────────────────────▼───────────┐ │
│  │              内嵌浏览器 (WebContentsView)                      │ │
│  │                                                              │ │
│  │  ┌─────────────────────────────────────────────────────────┐ │ │
│  │  │          小红书网页 (www.xiaohongshu.com)                 │ │ │
│  │  │                                                         │ │ │
│  │  │  ┌─── Extension Content Script ──────────────────────┐  │ │ │
│  │  │  │                                                   │  │ │ │

#### 6.1.3 三层数据提取策略

灵薯智作的采集引擎采用三层递进策略，确保数据提取的完整性和稳定性：

**Layer 1：`__INITIAL_STATE__` 静态数据提取（最快、最稳定）**

小红书网页在首次加载时，会将笔记数据注入到一个全局 JavaScript 变量 `__INITIAL_STATE__` 中。Content Script 可以直接从 `<script>` 标签或 `window.__INITIAL_STATE__` 中提取结构化 JSON 数据。

```
提取流程：
1. Content Script 在 document_idle 时机执行
2. 读取 window.__INITIAL_STATE__ 对象
3. 从中解析 noteData / userInfo / imageList 等字段
4. 每张图片的 imageList 中包含：
   - fileId  → 带水印图片 ID
   - traceId → 无水印原图 ID
   - url     → CDN 完整地址
   - width / height → 图片尺寸
5. 将结构化数据通过 chrome.runtime.sendMessage 发送
```

**数据字段映射：**

| `__INITIAL_STATE__` 路径 | 含义 | 提取方式 |
|--------------------------|------|---------|
| `note.noteDetailMap[id].note.title` | 笔记标题 | 直接取值 |
| `note.noteDetailMap[id].note.desc` | 笔记正文 | 直接取值 |
| `note.noteDetailMap[id].note.imageList` | 图片列表 | 遍历提取 |
| `note.noteDetailMap[id].note.video` | 视频信息 | 条件取值 |
| `note.noteDetailMap[id].note.interactInfo` | 互动数据 | 直接取值 |
| `note.noteDetailMap[id].note.tagList` | 标签列表 | 遍历提取 |
| `note.noteDetailMap[id].note.user` | 作者信息 | 直接取值 |

**Layer 2：DOM 解析 + MutationObserver（处理动态加载内容）**

当用户在信息流中滚动浏览、或 `__INITIAL_STATE__` 数据不完整时，需要实时解析页面 DOM：

```
提取流程：
1. 通过 CSS Selector 定位笔记卡片容器
   - 信息流：section.note-item / div[data-note-id]
   - 详情页：div.note-detail-container
2. 从 DOM 元素中提取文本、图片 src、data-* 属性
3. MutationObserver 监听 DOM 子树变化：
   - childList: true   → 监听新笔记卡片插入
   - subtree: true     → 监听嵌套内容更新
4. 新节点插入时自动触发解析逻辑
5. 去重：通过 noteId 判断是否已采集
```

**Layer 3：XHR / Fetch 拦截（捕获最完整的 API 原始数据）**

拦截小红书前端发出的 AJAX 请求，直接获取后端返回的 JSON 数据：

```
实现方式一：Content Script 中 Hook 原生 API
┌──────────────────────────────────────────────────┐
│  // 保存原始方法引用                                │
│  const originalFetch = window.fetch;              │
│  const originalXHROpen = XMLHttpRequest.prototype  │
│                          .open;                    │
│                                                    │
│  // 拦截 fetch                                     │
│  window.fetch = async (...args) => {               │
│    const response = await originalFetch(...args);   │
│    const url = args[0];                            │
│    if (url.includes('/api/sns/web/v1/feed'))        │

│    if (url.includes('/api/sns/web/v1/user'))        │
│      → 博主信息数据                                 │
│    // clone() response 并解析 JSON                  │
│    return response;                                │
│  };                                                │
└──────────────────────────────────────────────────┘

实现方式二：Electron 主进程通过 CDP 网络层拦截
┌──────────────────────────────────────────────────┐
│  // 通过 debugger 附加 CDP 协议                     │

**小红书核心 API 端点：**

#### 6.1.4 图片去水印技术

小红书图片去水印的核心利用了 CDN 系统中 `fileId` 和 `traceId` 的区别：

```
带水印图片 URL：
https://ci.xiaohongshu.com/{fileId}?imageView2/2/w/1080/format/webp

无水印原图 URL：
https://ci.xiaohongshu.com/{traceId}?imageView2/2/w/1080/format/webp
                             ▲
                             │
                    将 fileId 替换为 traceId
```

**完整的去水印流程：**

```
┌─────────────────────────────────────────────────────┐
│                  去水印处理管线                        │
│                                                     │
│  1. 从 __INITIAL_STATE__ 或 API 响应中提取 imageList │
│                                                     │
│  2. 对每张图片获取 traceId                            │
│     ┌──────────────────────────────────────────┐    │
│     │ imageList[i] = {                         │    │
│     │   url: "https://ci.xhs.com/abc123...",   │    │
│     │   fileId: "abc123...",    ← 带水印       │    │
│     │   traceId: "xyz789...",  ← 无水印原图     │    │
│     │   width: 1080,                           │    │
│     │   height: 1440                           │    │
│     │ }                                        │    │
│     └──────────────────────────────────────────┘    │
│                                                     │
│  3. 构造无水印 URL                                    │
│     将 url 中的 fileId 替换为 traceId                 │
│     可选去掉 imageView2 参数获取未压缩原图             │
│                                                     │
│  4. 在主进程通过 net/https 模块下载原图                │
│     携带必要的 Cookie 和 Referer 头                   │
│                                                     │
│  5. 保存到本地文件系统                                 │
│     路径: {userData}/media/images/{noteId}/{index}.jpg│
│                                                     │
│  6. 将本地路径写入 SQLite 数据库                       │
└─────────────────────────────────────────────────────┘
```

**视频去水印方案：**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 视频源地址直取 | 从 `__INITIAL_STATE__` 的 video 字段中获取 originVideoKey，拼接 CDN 域名下载无水印源视频 | 大多数情况 |
| 流媒体解析 | 解析 HLS (.m3u8) 或 DASH 分片地址，合并下载 | 长视频、高清视频 |
| FFmpeg 后处理 | 裁剪水印区域或使用 delogo 滤镜 | 源地址无法获取无水印版本时的兜底方案 |

#### 6.2.3 RPA 仿真引擎（核心执行层）

RPA 引擎是发布功能的核心执行组件，负责在内嵌浏览器中模拟真人操作。采用 **非侵入式仿真技术**，通过 CDP（Chrome DevTools Protocol）或 Electron WebContents API 发送底层输入事件。

**两种技术路线对比：**

| 维度 | Electron WebContents API | Chrome DevTools Protocol (CDP) |
|------|--------------------------|-------------------------------|
| 鼠标事件 | `webContents.sendInputEvent({ type: 'mouseDown' })` | `Input.dispatchMouseEvent({ type: 'mousePressed' })` |
| 键盘事件 | `webContents.sendInputEvent({ type: 'keyDown' })` | `Input.dispatchKeyEvent({ type: 'keyDown' })` |
| 文件上传 | 需要额外处理 | `DOM.setFileInputFiles` 直接注入文件路径 |
| 精确度 | 中等 | 高（可精确控制坐标、修饰键等） |
| 复杂度 | 低 | 中等 |
| 推荐度 | 简单场景 | 复杂交互场景（推荐） |

**核心操作的 RPA 实现细节：**

**(1) 鼠标移动 — 贝塞尔曲线拟人化**

```
直线移动 (机器人行为，容易被检测)：
  A ────────────────────────────▶ B

贝塞尔曲线移动 (模拟人手自然抖动)：
  A ───╮                   ╭───▶ B
       ╰──╮           ╭──╯
          ╰───╮   ╭──╯
              ╰──╯

实现原理：
  1. 起点 A(x1,y1) 到终点 B(x2,y2)
  2. 随机生成 2-3 个控制点，偏离直线路径
  3. 将曲线等分为 20-50 个采样点
  4. 每个采样点间隔 5-15ms 发送 mouseMoved 事件
  5. 速度曲线：慢启动 → 加速 → 减速 → 慢停止
  6. 添加 ±1-3px 的随机微抖动
```

**(2) 文字输入 — 逐字符拟人化输入**

```
输入文本: "今日穿搭分享"

时间轴 (ms):
  0    [今]    ← keyDown + keyUp
  120  [日]    ← 间隔 120ms (随机 80-200ms)
  235  [穿]    ← 间隔 115ms
  380  [搭]    ← 间隔 145ms
  550  [分]    ← 间隔 170ms（稍有犹豫）
  680  [享]    ← 间隔 130ms

关键实现细节：
  - 中文输入通过 CDP Input.insertText 逐字插入
  - 英文/数字通过 Input.dispatchKeyEvent 模拟按键
  - 随机偶尔插入 100-500ms 的「思考停顿」
  - 极低概率（1-2%）模拟打错字再删除的行为
```

**(3) 图片上传 — 文件注入**

```
┌─── 图片上传 RPA 流程 ────────────────────────────────┐
│                                                      │
│  方案 A: CDP DOM.setFileInputFiles (推荐)             │
│  ┌──────────────────────────────────────────────┐    │
│  │ 1. 定位 <input type="file"> 元素              │    │
│  │    CSS Selector: 'input[type="file"]'         │    │
│  │    或 XPath 定位隐藏的 file input              │    │
│  │                                              │    │
│  │ 2. 通过 CDP 注入文件路径                       │    │
│  │    Runtime.evaluate → 获取 input 的 nodeId     │    │
│  │    DOM.setFileInputFiles({                     │    │
│  │      files: ['/path/to/image1.jpg',            │    │

│  │      nodeId: inputNodeId                       │    │
│  │    })                                          │    │
│  │                                              │    │
│  │ 3. 触发 change 事件                           │    │
│  │    页面 JS 监听到文件变化，开始上传             │    │
│  │                                              │    │

**(4) 点击操作 — 精确元素定位与点击**

```
元素定位策略（按优先级）：
  1. data-* 属性选择器（最稳定）
     '[data-testid="publish-btn"]'
  2. 文本内容匹配
     找到包含"发布"文字的 button 元素
  3. CSS class 选择器（小红书改版可能变化）
     '.publish-btn', '.submit-button'
  4. XPath 相对定位
     '//button[contains(text(), "发布")]'

点击执行流程：
  1. 通过选择器获取元素的 boundingRect
  2. 计算元素中心点坐标 (cx, cy)
  3. 添加随机偏移 (cx ± random(3-8), cy ± random(3-8))
  4. 鼠标从当前位置以贝塞尔曲线移动到目标
  5. 停顿 50-150ms
  6. 发送 mousePressed → 停顿 80-120ms → mouseReleased
```

**node-schedule 使用方式：**

```

#### 6.2.7 错误处理与重试机制

│                                                            │

│  │                                                       │ │

│  │  Level 3: 不可恢复                                     │ │
│  │  ├── 账号被封禁 → 标记账号为 banned，停止所有任务       │ │
│  │  ├── 素材文件丢失 → 标记任务失败，通知用户               │ │
│  │  └── 页面结构重大变化 → 标记任务失败，等待软件更新       │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  ┌─── 重试策略 ──────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  指数退避 + 抖动 (Exponential Backoff with Jitter)     │ │
│  │                                                       │ │
│  │  重试次数:  1      2       3       (最多 3 次)         │ │
│  │  基础延迟:  30s    120s    300s                        │ │
│  │  随机抖动:  ±15s   ±60s    ±150s                      │ │
│  │  实际延迟:  15-45s 60-180s 150-450s                   │ │
│  │                                                       │ │
│  │  每次重试前重新检查：                                   │ │
│  │  - Session 是否仍然有效                                │ │
│  │  - 素材文件是否完整                                    │ │
│  │  - 是否有更高优先级任务需要先执行                       │ │
│  │                                                       │ │
│  └───────────────────────────────────────────────────────┘ │
│                                                            │
│  ┌─── 日志记录 ──────────────────────────────────────────┐ │
│  │                                                       │ │
│  │  publish_logs 表:                                      │ │
│  │  ┌────┬────────┬──────┬─────────┬───────────────────┐ │ │
│  │  │ id │ 任务ID │ 级别 │ 时间     │ 详情               │ │ │
│  │  ├────┼────────┼──────┼─────────┼───────────────────┤ │ │
│  │  │ 1  │ t_001  │ INFO │ 10:30:00│ 开始执行发布       │ │ │

### 6.5 多账号管理

#### 技术方案

```
┌───────────────────────────────────────┐
│          账号管理模块                   │
│                                       │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ │
│  │ 账号 A   │ │ 账号 B   │ │ 账号 C   │ │
│  │ Session  │ │ Session  │ │ Session  │ │
│  │ Cookie   │ │ Cookie   │ │ Cookie   │ │
│  │ Profile  │ │ Profile  │ │ Profile  │ │
│  └────┬────┘ └────┬────┘ └────┬────┘ │
│       │          │          │        │
│       ▼          ▼          ▼        │
│  每个账号独立的 Electron Session      │
│  (partition / BrowserSession)        │
│  隔离 Cookie、Cache、Storage         │
└───────────────────────────────────────┘
```

- 使用 Electron 的 `session.fromPartition()` 为每个账号创建独立会话
- 各账号的 Cookie / LocalStorage 完全隔离
- 扫码登录：在内置浏览器中展示小红书登录二维码，用户手机扫码完成登录

### 6.6 自动更新机制

- 使用 `electron-updater` 实现自动检查和下载更新
- 公测阶段高频更新，增量更新减少下载量
- 可能使用差分更新 (delta update) 优化更新体验

## 7. 若从零开发的技术选型建议

### 7.1 桌面应用框架

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Electron** | 生态成熟、Web 技术栈、内置 Chromium 完美支持 Extension/RPA | 内存占用大、包体积大 | ⭐⭐⭐⭐⭐ |
| Tauri 2.0 | 包体积小、性能好、Rust 安全性 | 内嵌浏览器控制能力弱、Extension 支持差 | ⭐⭐ |
| CEF (C++) | 性能极致、浏览器控制能力强 | 开发成本高、跨平台复杂 | ⭐⭐⭐ |

**推荐：Electron**，因为此类产品的核心是内嵌浏览器 + Extension 注入 + RPA 自动化，Electron 基于 Chromium 天然适合。

### 7.2 前端技术栈

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| UI 框架 | React 18/19 + TypeScript | 成熟生态，组件丰富 |
| 样式方案 | Tailwind CSS | 快速开发，原子化 CSS |
| 状态管理 | Zustand 或 Jotai | 轻量、适合 Electron 场景 |
| 路由 | React Router 或 TanStack Router | 桌面端多视图管理 |
| 构建工具 | Vite + electron-vite | 极速 HMR，Electron 集成好 |
| UI 组件库 | Ant Design / Radix UI | 数据表格等复杂组件 |
| 图表库 | ECharts / Recharts | 数据看板可视化 |
| 图标 | Lucide React | 从源码确认使用了 Lucide 图标 |

### 7.3 Electron + 浏览器自动化

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| Electron 版本 | 最新稳定版 | 保持 Chromium 版本更新 |
| 内嵌浏览器 | BrowserView / WebContents | 加载目标网站 |
| Extension 注入 | `ses.loadExtension()` | Electron 原生支持加载 Chrome Extension |
| RPA 事件模拟 | CDP (Chrome DevTools Protocol) | `Input.dispatchMouseEvent` 等 |
| 页面解析 | Content Script (Extension) | 注入到目标页面的 JS 脚本 |
| 元素定位 | CSS Selector / XPath | 定位页面元素 |

### 7.4 本地数据存储

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| 数据库 | better-sqlite3 | Electron 中使用 SQLite 的最佳方案 |
| ORM | Drizzle ORM / Kysely | 类型安全的 SQL 查询构建 |
| 加密 | Node.js crypto + OS Keychain | AES 加密 + keytar 库访问系统密钥链 |
| 文件存储 | electron app.getPath('userData') | 素材文件本地存储 |
| 配置 | electron-store | 简单键值对配置持久化 |

### 7.5 AI 集成

| 组件 | 推荐方案 | 说明 |
|------|---------|------|
| AI SDK | 各平台官方 SDK 或直接 HTTP 调用 | 火山引擎 SDK、阿里百炼 SDK |
| Prompt 管理 | 模板引擎 (Handlebars / 自研) | 管理不同功能场景的 Prompt 模板 |
| 流式输出 | SSE (Server-Sent Events) | 实现打字机效果的实时输出 |
| 备选方案 | OpenAI 兼容接口 | 火山引擎和百炼均兼容 OpenAI 接口格式 |

### 7.6 后端服务（轻量）

| 组件 | 推荐方案 | 说明 |

## 8. 风险与合规考量

### 8.1 平台政策风险

| 风险 | 说明 | 应对策略 |
|------|------|---------|
| 小红书反爬 | 平台可能升级反爬机制，导致采集失效 | RPA 仿真 + 行为随机化 + 高频更新适配 |
| 账号封禁 | 批量操作可能触发风控 | 控制操作频率、模拟真人行为、用户教育 |
| 平台 TOS 违规 | 自动化操作可能违反小红书用户协议 | 法务评估、用户免责声明 |

### 8.2 数据安全风险

| 风险 | 说明 | 应对策略 |
|------|------|---------|
| 本地数据泄露 | 设备被盗或恶意软件窃取 | 本地加密 + 系统级密钥链 |
| API Key 泄露 | 用户 API Key 被截获 | 加密存储、HTTPS 传输 |

### 8.3 法律合规

| 维度 | 说明 |
|------|------|
| 内容版权 | 采集他人内容涉及版权，需提醒用户合规使用 |
| 数据隐私 | 采集博主公开数据一般合规，但需遵守个人信息保护法 |
| AI 生成内容 | AI 仿写需确保不构成抄袭，建议用户二次修改 |

### 8.4 技术维护成本

| 维度 | 说明 |
|------|------|
| 小红书改版适配 | 页面 DOM 结构变化需要及时更新 Extension |
| Electron 安全更新 | Chromium 安全漏洞需要及时升级 |
| 大模型 API 变更 | 第三方 API 接口变更需要适配 |
| 多平台兼容 | macOS / Windows 双平台构建和测试 |
